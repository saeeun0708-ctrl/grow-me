# 나를 키워줘 — 단계별 개발 프롬프트 (7단계)

> 사용법: 각 단계의 "프롬프트" 블록을 순서대로 실행한다.
> 한 단계가 끝나면 **완료 기준(DoD)** 을 체크하고 다음 단계로 넘어간다.
> 모든 단계는 이전 단계 결과 위에 쌓인다.

---

## 단계 개요

| 단계 | 제목 | 한 줄 목표 |
|---|---|---|
| 1 | 프로젝트 셋업 & 디자인 시스템 | 모바일 웹 골격 + 더미 데이터로 화면 흐름 확인 |
| 2 | Supabase 연동 & 간편 로그인 | DB·인증 붙이고 내 캐릭터 영속화 |
| 3 | 먹이주기 플로우 (지인용) | 이름 입력 + 제시어 7개 선택 + 저장 |
| 4 | 성장 로직 & 캐릭터 결정 | 단계 계산 + 8종 캐릭터 확정 + 주인 화면 |
| 5 | 결과 화면 & 이미지 저장 | 1~5위 랭킹 + 인스타용 카드 PNG + 워터마크 |
| 6 | 수익화 (상세 언락 결제) | 페이월 + 토스페이먼츠 결제 + 이름별 상세 |
| 7 | 바이럴 마감 & 배포 | 독립시키기 + OG태그 + 공유 + 배포 |

---

## 1단계 — 프로젝트 셋업 & 디자인 시스템

**목표**: 데이터 없이도 전체 화면 흐름을 클릭으로 확인할 수 있는 프로토타입.

```프롬프트
docs/PRD.md를 기준으로 "나를 키워줘" 모바일 웹의 기본 골격을 만들어줘.

요구사항:
1. Vite + Vanilla JS(또는 가벼운 셋업)로 모바일 우선(세로 393px 기준) 프로젝트 생성.
2. 디자인 시스템: PRD.html의 색감(파스텔 퍼플/핑크), 둥근 모서리, 귀여운 톤을 CSS 변수로 정의.
3. 더미 데이터로 아래 화면을 라우팅(해시 라우터)으로 연결:
   - 랜딩/로그인 (로그인 버튼은 더미)
   - 내 캐릭터(주인) 화면 - 단계별 캐릭터 자리(임시 이모지/플레이스홀더), 받은 먹이 수, 공유 버튼
   - 먹이주기(지인) 화면 - 이름 입력칸 + 제시어 칩 40개 + 먹이주기 버튼(비활성/활성)
   - 결과 화면 - 1~5위 랭킹 더미
4. 제시어 데이터와 캐릭터-카테고리 매핑은 PRD 부록 A의 JSON을 src/data/words.js로 분리.
5. 캐릭터 이미지는 아직 없으니 placeholder(단계·코드 표시) 컴포넌트로.

지금은 백엔드 없이 더미로만. 화면 전환과 칩 7개 선택 제한(7개 채우면 버튼 활성)까지 동작하게.
```

**완료 기준(DoD)**
- [ ] 로컬에서 `npm run dev`로 실행, 네 화면 클릭 이동 가능
- [ ] 제시어 칩 7개 선택 시에만 먹이주기 버튼 활성화
- [ ] words.js에 40개 제시어 + 8 카테고리 매핑 존재

## 2단계 — Supabase 연동 & 간편 로그인

**목표**: 로그인하면 내 캐릭터가 DB에 생성·유지된다.

```프롬프트
Supabase를 연동하고 간편 로그인을 붙여줘.

1. Supabase 프로젝트 가정하에 .env로 URL/anon key 관리 (env 예시 파일 포함).
2. PRD 8장의 스키마대로 테이블 SQL 작성: users, feedbacks. (SQL 파일 docs/schema.sql로)
   - users.share_slug는 고유 랜덤 슬러그(6자) 자동 생성.
3. Supabase Auth로 카카오/구글 OAuth 로그인 구현. (둘 중 구글 먼저 동작, 카카오는 설정 가이드 주석)
4. 최초 로그인 시 users row 자동 생성(없으면 insert), stage=1, character_code=null.
5. 로그인 후 "내 캐릭터" 화면에서 내 share_slug 기반 공유 링크(/feed/:slug)를 보여주고 복사 버튼.
6. 인증 상태에 따라 라우팅 가드(비로그인은 랜딩으로).

기존 더미 화면을 실제 user 데이터에 연결하되, 먹이 카운트는 아직 0으로.
```

**완료 기준(DoD)**
- [ ] 구글 로그인 → users row 생성 확인(Supabase 콘솔)
- [ ] 내 공유 링크 `/feed/{slug}` 복사 가능
- [ ] 새로고침·재로그인해도 내 캐릭터 유지

## 3단계 — 먹이주기 플로우 (지인용)

**목표**: 지인이 링크로 들어와 먹이주기를 완료하면 feedbacks에 저장된다.

```프롬프트
지인용 먹이주기 플로우를 완성해줘. (로그인 불필요)

1. /feed/:slug 접속 시 해당 slug의 주인 정보 로드. 없으면 안내 화면.
2. "○○님에게 먹이주기" 헤더 + 이름 입력 + 제시어 칩 40개(7개 선택 제한).
3. [먹이주기] 누르면 feedbacks insert: owner_user_id, sender_name, selected_words(7개).
4. 중복 방지: localStorage에 제출한 slug 기록 → 같은 기기로 같은 주인에게 재제출 막기(안내 메시지).
5. 완료 화면: 성장 애니메이션(간단한 CSS) + "○○님 캐릭터가 성장했어요!" + "나도 만들기" CTA(랜딩으로).
6. insert 후 주인의 stage/character_code를 서버에서 재계산(다음 단계 로직과 연결 지점만 마련).

모바일에서 엄지로 3탭 내 완료되게 단순하게.
```

**완료 기준(DoD)**
- [ ] 링크 접속 → 이름+7개 선택 → 저장 성공
- [ ] feedbacks 테이블에 row 적재 확인
- [ ] 같은 기기 재제출 차단 동작
- [ ] 완료 화면에 "나도 만들기" CTA

## 4단계 — 성장 로직 & 캐릭터 결정

**목표**: 먹이 수와 제시어 집계로 단계·캐릭터가 정확히 결정된다.

```프롬프트
성장/캐릭터 결정 로직을 구현해줘.

1. 집계 함수: 특정 owner의 feedbacks를 모아 selected_words를 펼치고,
   PRD 부록 A 매핑으로 카테고리별 합계 계산.
2. 캐릭터 결정: 합계 1위 카테고리 = character_code. 동점이면 먼저 1위 도달(타임스탬프) 우선.
3. 단계 결정: feedback 수 0→1단계, 1~2→2단계, 3+→3단계.
4. 이 계산을 Supabase RPC(혹은 클라이언트 계산 후 users 업데이트)로 만들고,
   먹이 insert 직후 users.stage / character_code 갱신.
5. "내 캐릭터" 화면에 실제 단계별 캐릭터 placeholder + 받은 먹이 수 + (3단계 시) 독립 버튼 노출.
6. 단위 테스트 시나리오 주석: 먹이 0/1/3/여러개 케이스별 기대 결과.

캐릭터 이미지는 src/assets/characters/{code}_{stage}.png 자리만 잡고 placeholder 유지.
```

**완료 기준(DoD)**
- [ ] 먹이 1개 → 2단계, 3개 → 3단계 전환 확인
- [ ] 제시어 분포에 따라 8종 중 올바른 캐릭터로 확정
- [ ] 동점 처리 규칙 동작

## 5단계 — 결과 화면 & 이미지 저장

**목표**: 자랑하고 싶은 결과 카드를 PNG로 저장한다. (바이럴 엔진)

```프롬프트
결과 화면과 인스타용 이미지 저장을 구현해줘.

1. 결과 화면(무료 영역): 확정 캐릭터, 1~5위 제시어 + 각 받은 수, 총 먹이 수, 캐릭터 타입명.
2. html-to-image로 결과 카드를 PNG 생성. 두 사이즈 선택: 스토리 1080×1920 / 피드 1080×1350.
3. 카드 디자인: 캐릭터 크게, 1~5위 랭킹, 받은 수, 하단에 "나를 키워줘" 워터마크+로고+짧은 URL.
4. [이미지 저장] 버튼 → 모바일에서 다운로드/공유 시트 호출.
5. 캐릭터 이미지 에셋이 들어오면 교체될 수 있게 컴포넌트화.

캡처가 아니라 앱이 직접 고해상도 카드를 렌더링해서 저장되도록.
```

**완료 기준(DoD)**
- [ ] 결과 카드가 정확한 인스타 비율로 PNG 저장됨
- [ ] 워터마크·로고·URL 포함
- [ ] 1~5위와 받은 수가 실제 데이터와 일치

## 6단계 — 수익화 (상세 언락 결제)

**목표**: "누가 무슨 제시어를 골랐는지"를 결제로 언락한다.

```프롬프트
상세 언락 결제(페이월)를 구현해줘.

1. 결과 화면 하단에 잠금 영역: "누가 나를 OO라고 했을까? 👀" + 블러 처리된 이름 리스트.
2. [전체 보기 결제] → 토스페이먼츠(테스트 키) 결제창. 성공 시 users.is_unlocked = true.
   - 결제 검증은 서버리스 함수(또는 Supabase Edge Function)로 결제 승인 처리.
3. is_unlocked = true면:
   - 전체 제시어 분포(6위 이하 포함)
   - sender_name별로 어떤 제시어를 골랐는지 상세 리스트
4. 가격/문구는 PRD 7장 기준(₩990). 결제 실패·취소 처리 포함.
5. 이미 언락한 사용자는 재진입 시 바로 상세 노출.

테스트 결제로 전체 플로우 검증 가능하게.
```

**완료 기준(DoD)**
- [ ] 테스트 결제 성공 → is_unlocked 전환
- [ ] 언락 후 이름별 상세·전체 분포 노출
- [ ] 결제 취소/실패 시 안전하게 원복

## 7단계 — 바이럴 마감 & 배포

**목표**: 공유 최적화 + 독립시키기 + 실제 배포.

```프롬프트
바이럴 요소를 마감하고 배포 준비를 해줘.

1. 독립시키기: 3단계에서 [독립시키기] → 확인 모달 → users.is_independent=true.
   이후 해당 slug 먹이주기 화면은 "성장이 끝난 캐릭터예요" 안내로 막고 결과 고정.
2. OG/트위터 메타태그: 공유 링크 미리보기에 캐릭터·문구 노출(슬러그별 동적 OG는 서버리스로).
3. 공유 버튼: 링크 복사 / 카톡 공유 / 결과 이미지 저장을 한 시트에 정리.
4. "나도 만들기" CTA를 지인 완료 화면·결과 카드 근처에 일관 배치.
5. 반응형·로딩·에러 상태 점검, 라이트 성능 최적화.
6. Vercel(또는 Netlify) 배포 설정 + 환경변수 가이드 문서화.

마지막으로 전체 사용자 여정(가입→공유→먹이→성장→결과→결제→독립)을 한 번 테스트.
```

**완료 기준(DoD)**
- [ ] 독립시키기 후 먹이 차단·결과 고정
- [ ] 공유 링크 OG 미리보기 정상
- [ ] 배포 URL에서 전체 여정 동작

## 별도 트랙 — 캐릭터 에셋 (클로드 디자인)

개발과 병행. PRD 5.2의 특징 정의서를 클로드 디자인에 전달해 8종 × 3단계 = 24종 + 공통 1단계 알 제작.
파일명 규칙: `assets/characters/{code}_{stage}.png` (예: `brain_3.png`).
2단계는 3단계 특징이 절반만 드러난 형태로 통일.
